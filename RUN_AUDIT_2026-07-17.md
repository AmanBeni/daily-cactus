## 1. Google News URL resolution (HIGHEST IMPACT)

**Flaw found.** About half of every section's candidates arrive via Google
News RSS queries (`sources.yaml`). Their `<link>` is a
`news.google.com/rss/articles/CBMi...` redirect, not the publisher URL. That
broke two things: (a) `enrich_shortlist.py` fetched Google's own interstitial
page instead of the article, so full-text extraction silently failed for
~half the digest; (b) the reader's "Read the original" link sent them to a
Google interstitial instead of the article.

**Fix.** New `scripts/resolve_gnews.py`, `resolve(url) -> str`:
1. Cheap offline attempt: base64-decode the article-id path segment and
   regex out an embedded `http(s)://` URL (works for older-style tokens).
2. Network fallback: fetch the redirect page with a real Chrome UA, extract
   its `data-n-a-id` / `data-n-a-ts` / `data-n-a-sg` triple, and POST that to
   Google's own internal `news.google.com/_/DotsSplashUi/data/batchexecute`
   endpoint (the same call the Google News web app itself makes) to get the
   real URL back. Result is cached within the run.
3. Never raises — any failure at any stage returns the **original** url
   unchanged, so a resolution miss degrades exactly like before this fix,
   never crashes the pipeline.

Wired into `build_digest.py::shortlist_section`, resolving each story's URL
**after** shortlisting/diversity-capping (only the final ≤~180 survivors per
run, not the raw ~500 candidates) and **before** `make_id`/`refs` are built —
so the id, the refs snapshot (and therefore both enrichment and the reader
link), and cross-day dedup all key off the real URL.

**Verified live** — 3 real `news.google.com/rss/articles/...` URLs pulled
from a live `fetch_feeds.py` run, resolved to real publisher domains:
- Nous Research funding story → `techcrunch.com/2026/07/13/hermes-agent-maker-nous-research-in-talks-for-new-funding-at-1-5b-valuation/`
- Argentina World Cup story → `indiatoday.in/sports/football/story/lionel-messi-argentina-fifa-world-cup-...`
- Stem-cell therapy story → `insideprecisionmedicine.com/topics/translational-research/stem-cell-therapy-advances-chronic-spinal-cord-injury/`

Broader spot-check: 8/8 sampled Google News URLs across sections resolved
successfully (~1.6s/url). In the full live rebuild (see §3), **0 of 148**
refs entries still point at `news.google.com` — every Google News link in
today's digest resolved to a real publisher domain.

## 2. Browser User-Agent + retry in enrichment

**Flaw found.** `enrich_shortlist.py` sent a bot-flavoured UA
(`DailyCactusBot/1.0`, or trafilatura's own default fetcher) instead of a
browser UA. Verified: Indian Express blocks the bot UA outright.

**Fix.** Centralized fetching in a new `_fetch_html(url)` helper: a full,
current Chrome UA (`Chrome/126.0.0.0`), one retry on any failure, then
**both** `trafilatura.extract()` and `readability.Document()` parse that same
already-fetched HTML (previously each extractor fetched independently with
its own UA; now there's one fetch, shared).

**Verified live**:
```
python3 scripts/enrich_shortlist.py "https://indianexpress.com/article/india/trump-administration-us-student-journalists-visa-rules-fixed-term-10790151/"
```
Returns real article text (2500 chars, the new cap) containing **"four
years"** — confirmed via direct string check.

## 4. Prompt guard against padding (`ROUTINE_PROMPT.md`)

**Fix.** Added a charter rule: with no `extract` (headline + short RSS blurb
only), write a **shorter**, honest summary of just what's known — never pad
with assumed background or speculation (example given: never invent "this
typically signals an IPO"). Numbers/claims must come from the digest text,
never assumed knowledge. Cross-referenced from the `summary` field spec to
avoid restating it twice.

Trimmed the "Honesty about thin days" bullet (dropped the redundant
"never web-search" clause, already stated in the read-instructions section)
to keep the file roughly the same length: **8,186 → 8,433 bytes** (+3%).

## 5. Broken feeds (`sources.yaml`)

Tested all 5 live with feedparser (using a correct CA bundle — see the
generic fix below, which was itself needed to get an honest read):

| Feed | Verdict | Action |
|---|---|---|
| `feeds.hbr.org/harvardbusiness` | Real, reproducible `SSL: UNEXPECTED_EOF_WHILE_READING` (5/5 attempts) | **Kept** (owner call: may be transient CDN issue) — relabeled `[FLAKY]` with a note to revisit via the monthly feed audit if the hit rate stays at 0. |
| `importai.substack.com/feed` | **Confirmed working** (20-100+ entries in repeated tests) | No change — likely a one-off transient failure in that specific run. |
| `inc42.com/industry/deeptech/feed/` | **Genuinely dead** — 301-redirects to the HTML category page; Inc42 renamed the slug to hyphenated `deep-tech` | **Fixed**: corrected URL to `inc42.com/industry/deep-tech/feed/` — confirmed 24 entries. |
| `downtoearth.org.in/rss/agriculture` | **Genuinely dead** — DTE retired per-category feeds (301s to HTML; no RSS `<link>` on the page either; their only surviving feed is an all-topics `/stories.rss`) | **Replaced** with a verified India-agriculture Google News query (`agriculture OR farmers OR "crop prices" OR MSP OR irrigation OR "farm policy" OR drought OR monsoon OR "farm income" + India, when:4d`) — confirmed 100 entries incl. The Hindu, India Today, ETV Bharat, and Down To Earth itself (via aggregation). |
| `nature.com/nature.rss` | **Intermittently flaky**, root-caused (see below) | Fixed generically (see below); relabeled `[FLAKY]`, kept. |

**Root cause found for the "invalid XML" pattern** (nature.com, and likely
others): it wasn't malformed XML. A/B test — fetching the identical URL with
plain `urllib` and handing the raw bytes to `feedparser.parse(data)` parsed
cleanly (75 entries) every time; letting `feedparser.parse(url)` do its own
fetch failed intermittently with a "not well-formed" error at the exact byte
position of a multi-byte UTF-8 "©" character. That points at feedparser's own
gzip-aware fetcher mis-handling a compressed response from certain CDNs
(nature.com sits behind Fastly/Varnish, whose response varies by
`Accept-Encoding`). **Fix**: `fetch_feeds.py` now fetches every feed's raw
bytes itself via `urllib` (no compression requested) and hands them to
`feedparser.parse(data)` instead of `feedparser.parse(url)` — a structural
fix that helps any feed hitting this class of bug, not just nature.com.
Falls back to feedparser's own url-fetch if the manual request raises, so
existing redirect/edge-case handling isn't lost.

Also hardened: `fetch_feeds.py` now forces a current, complete CA bundle
(`certifi`) as the process-wide default HTTPS context, so a feed's real
health is never confounded by a runner's trust-store quirks masquerading as
an "invalid XML" bozo error.

**Section feed counts, before → after** (no explosion — 1-for-1 swaps):
- Agritech: 4 → 4 feeds
- India Deep Tech: 3 → 3 feeds
- Global Economics: 5 → 5 (hbr kept)
- Other Interests: 9 → 9 (nature.com kept)

**Live re-fetch after all §5 fixes**: `Feeds OK: 58/61` (up from 56/61),
568 entries. Remaining failures: `feeds.hbr.org` (real SSL EOF, kept by

the next real Actions run rather than "fixed" against a guess.

## 6. Frontpage/section duplication (`site/app.js`)

**Flaw found.** A story chosen for the front page that also lived in its
home section rendered twice — a full lead/frontpage card, then a
near-identical full card again under its section.

**Fix.** In `renderEdition`, section-story rendering now checks each story's
`url` against the front-page anchor map already built in Pass 1
(`urlAnchors`, keyed by url, registered lead/frontpage-first so a shared url
resolves to the front-page anchor even when the section registers the same
url second as a no-op). On a match, renders a new compact one-line
`crossRefHTML()` ("↑ on the front page", linking to the front-page card)
instead of the full card. Back-compatible: editions/stories with no `url` (or
old editions) simply never match, so behavior is unchanged for them.

**Verified**: added a duplicate story to `site/dev-fixture.json` (the lead,
"Sarvam AI ... $234M HCLTech-led round", also placed in the Indian Startups
section alongside a genuinely unique story). Served `site/` locally
(`python3 -m http.server` + `?fixture=1`) and inspected the rendered DOM:
the Indian Startups section shows the Sarvam AI story as a one-line
"↑ on the front page" cross-reference (linking to `#2026-07-14/front/0`,
the lead's anchor) while the unique Zepto story renders as a full card,
confirmed via both the accessibility tree and extracted page text.

## 7. Quick checks

**Why 0 opportunities?** `drafts/2026-07-10.json` (the most recent draft
available in this working copy) has `"opportunities": []` — an explicit,
schema-valid empty list, not malformed output. Investigated the shortlist
that section receives: opportunities candidates come exclusively from Google
News queries, which (a) get `summary=""` by design (`fetch_feeds.py`'s
gnews branch: "gnews summaries are noisy link-lists") and (b) skew toward
retrospective *coverage* of events/conferences rather than forward calendar
announcements — e.g. today's shortlist includes "IndiaAI Mission selects 10
startups..." (a past-tense result, not an upcoming opportunity) and "INDIAN
TECH START-UP REPORT 2025" (a report, not an event). With only a bare
headline and the editor's own charter requiring "a concrete date/deadline
AND a link that has NOT already passed," the honest, charter-compliant
outcome on a day when nothing clears that bar is `[]` — which the prompt
explicitly sanctions ("If none qualify, use `[]`").

**Verdict: thin/structurally-noisy day, not an obvious code bug** — no fix
applied here beyond what's already indirect: Fixes 1+3 (resolved links +
2500-char extracts) mean the editor can now actually read an opportunity's
full text instead of a bare headline, which should surface real dates more
often going forward. If 0-opportunity days persist after this, the next
lever would be rewriting the Opportunities Google News queries to bias
toward "registration open" / "apply by" / "deadline" language — flagged for
the owner as a follow-up, not implemented here (out of the "fix only if
obvious bug" instruction).

**Secret printing check.** Grepped `.github/workflows/*.yml` and
`scripts/*.py` for anything that echoes a token/secret to stdout — found
nothing. The three workflows that use `secrets.GITHUB_TOKEN` pass it
directly into `actions/checkout`, `peaceiris/actions-gh-pages`, and
`actions/github-script`'s `github-token` inputs; none of our own code prints
it. The masked `token: ***` line the live fetch log showed is GitHub's own
automatic secret-masking on `actions/checkout`'s setup step — that's the
*safe*, expected behavior (GitHub redacts any string matching a registered
secret wherever it appears in a log), not a leak. No fix needed.

Separately (not in scope, flagged for awareness only): this working copy
has a `GITHUB_TOKEN.local.md` at the repo root containing a live-looking
`github_pat_...` token in plaintext. It IS covered by `.gitignore`
(`*.local.md`), so it can't leak via a commit from this repo — but it's
still a live credential sitting on disk. Worth the owner revoking it once
no longer needed, per that file's own instructions.
